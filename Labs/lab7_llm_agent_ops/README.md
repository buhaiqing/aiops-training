# Lab 7: LLM Agent 自主运维实战

## 🎯 实验目标

本实验模拟真实的运维场景，让 AI Agent 自主分析告警、诊断问题并给出解决方案。

### 核心能力展示

1. **多源告警数据融合** - 整合 Prometheus、Grafana、ELK、Zabbix 等多个监控系统的告警
2. **智能关联分析** - 识别告警之间的因果关系和时间序列模式
3. **根因推断** - 基于 LLM 的推理能力定位问题根源
4. **处置方案生成** - 自动生成可执行的修复命令和操作步骤

## 🤖 LLM Agent 核心能力

### 1. 告警关联分析

AI Agent 能够识别告警之间的关联性：

```python
# 输入：15 条独立告警
{
  "alert_id": "ALT-001",
  "metric": "node_cpu_usage",
  "value": 95,
  "host": "web-server-01",
  ...
}

# 输出：3 个关联事件组
{
  "host": "web-server-01",
  "alert_chain": [CPU 过载 → 内存压力 → 响应慢 → 错误率高],
  "potential_root_cause": "CPU 过载导致服务响应变慢",
  "time_span_minutes": 10
}
```

### 2. 根因推断

基于症状链进行因果推理：

```
观察到的现象:
1. CPU 使用率持续上升（无波动）
2. 伴随内存压力
3. 响应时间同步恶化
4. 错误率随后上升

LLM 推理:
✓ 排除外部流量冲击（因为增长平滑）
✓ 排除偶发故障（因为持续时间长）
✓ 推测存在死循环或资源泄漏
✓ 置信度：85%
```

### 3. 处置方案生成

自动生成可执行的操作步骤：

```markdown
P0 - 立即执行:
  动作：定位高负载进程
  命令：top -bn1 | head -20
  预期：识别占用 CPU 最高的进程

P1 - 5 分钟内:
  动作：临时降低优先级
  命令：renice +10 -p <PID>
  预期：缓解系统压力

P2 - 30 分钟内:
  动作：重启异常服务
  命令：systemctl restart nginx
  预期：恢复正常状态
```

### 4. 运维手册匹配

根据问题类型推荐标准操作流程：

```
检测到：CPU 过载
↓
匹配运维手册：RB-CPU-HIGH
↓
推荐关键步骤:
1. 确认告警真实性 (top)
2. 识别高负载进程 (ps aux)
3. 分析进程行为 (strace)
4. 临时缓解 (renice)
5. 重启服务 (systemctl)
```

## 🔬 技术原理

### 为什么不需要真实 LLM？

本实验采用**规则引擎 + 模板匹配**的方式模拟 LLM 的输出，原因：

1. **教学目的** - 展示 AI 决策的逻辑过程，而非黑盒
2. **可解释性** - 每一步推理都有明确的依据
3. **零成本** - 无需调用付费的 API
4. **可控性** - 输出质量稳定，不会胡编乱造

### 如何扩展到真实 LLM？

只需替换 `generate_diagnosis_report` 方法：

```python
def generate_diagnosis_report(self, correlations):
    # 构建 prompt
    prompt = f"""
    你是一位资深运维专家。请分析以下告警信息:
    
    {json.dumps(correlations, indent=2)}
    
    请提供:
    1. 根因分析
    2. 影响评估  
    3. 处置建议
    4. 优先级排序
    """
    
    # 调用 LLM API
    response = call_llm_api(prompt)  # OpenAI/Qwen/ChatGLM
    
    return response
```
